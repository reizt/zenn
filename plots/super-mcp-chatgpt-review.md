# 「Super MCP」記事の評価基準

## 採点式

合計100点で評価する。

| 評価軸 | 配点 | 満点の条件 |
| --- | ---: | --- |
| 読者価値と主張 | 15 | 誰のどの問題を扱い、読後に何を持ち帰れるかが冒頭で分かる |
| 背景と問題設定 | 10 | 解決策を先回りせず、実際に起きていた問題と原因を区別して示す |
| 構成と節の役割 | 15 | 各節が一つの問いに答え、同じ粒度の説明を別の節で繰り返さない |
| 技術説明 | 20 | 入力、変換、実行、認証の流れを具体例から追え、実装範囲も分かる |
| 設計判断 | 15 | 採用案だけでなく、代替案、選択理由、得たもの、失ったものを示す |
| 正確さと限定 | 10 | 仕様上の役割と実装事実を区別し、未計測、未実装、対象外を明記する |
| 図と実例 | 5 | 図が本文の理解を助け、実際の利用場面か検証結果が主張を裏付ける |
| 文章品質 | 5 | 丁寧語、一文一義、用語の一貫性、冗長さのなさを満たす |
| 会社ブログとしての完成度 | 5 | 会社と著者の文脈、ブログリレーの導入と結びが自然に収まる |

## 判定基準

- 90点以上：強い記事。固有の実装と判断が、実例を伴って読者へ伝わる
- 85〜89点：公開可能。大きな欠落はないが、実証か読み物としての吸引力に改善余地がある
- 80〜84点：技術草稿として良好。主張を裏付ける材料か構成に不足がある
- 79点以下：公開前に正確さまたは論証を直す必要がある

一つでも配点の60%未満の軸があれば、合計点にかかわらず公開前に修正する。

## 改善ループ

1. 会話の経緯を持たない三つの視点（初見読者、技術編集、仕様監査）で採点する
2. 複数の評価で共通した指摘と、正確さに関する指摘を優先する
3. 見込み加点が2点以上の変更だけを反映する
4. 同じ基準で再採点する
5. 90点以上、全軸60%以上、前回からの改善が2点未満になった時点で終了する

## 第1回評価

独立評価は82点、81点、73点だった。

共通していた減点は次の通り。

- ChatGPTから使った実例がなく、作った結果を検証できない
- OAuthでSuper MCPが担う役割の説明が不足している
- 構成図のChatGPTからの線がD1へ接続して見える
- tool数を固定した効果を計測しておらず、設計の評価が途中で止まっている
- 任意URLを取得することによるSSRFのリスクが弱く扱われている

統一基準へ換算した初回スコアは79点とする。

| 評価軸 | 得点 |
| --- | ---: |
| 読者価値と主張 | 12/15 |
| 背景と問題設定 | 9/10 |
| 構成と節の役割 | 12/15 |
| 技術説明 | 15/20 |
| 設計判断 | 12/15 |
| 正確さと限定 | 6/10 |
| 図と実例 | 3/5 |
| 文章品質 | 5/5 |
| 会社ブログとしての完成度 | 5/5 |

## 第1回で反映した変更

- MCPに対するOAuth認可サーバーとリソースサーバーの役割を区別した
- Dynamic Client Registrationが必須要件ではないことを明記した
- 二つのOAuthシーケンス図を実際のリダイレクトとtoken更新条件に合わせた
- 対応するOpenAPIの入力形式と、扱わない形式を明記した
- annotationの推定をOpenAPI変換の節へ移し、権限制御ではないと明記した
- tool数を固定した効果は未計測であると明記した
- SSRFを個人利用に限定する主要因として説明した
- 構成図の接続線とOAuthの役割表記を修正した

## 未解決

実際の利用例には、登録済みのHevy APIを使ってトレーニング履歴を振り返る場面が適している。

天気APIは動作確認にはなるが、Super MCPの有用性を示す例として弱いため採用しない。

個人のトレーニングデータを含むため、本文へ載せる画面と値は著者が公開範囲を決めてから追加する。

## 第2回評価

初回と異なる三つのサブエージェントが、同じ評価表で96点、87点、82点を付けた。

平均は88.3点だったが、仕様監査では「正確さと限定」が3/10となり、公開条件を満たさなかった。

実装とMCP仕様を再照合し、次の不整合を確認した。

- 認可requestの`resource`が任意で、token requestとaccess tokenのaudience bindingへ反映されていない
- PKCEなしと`plain`方式を許可している
- Dynamic Client Registrationでredirect URIを十分に制限していない
- ログイン済み利用者へ明示的な同意画面を出していない
- `list_commands`と`call_command`が`outputSchema`を宣言しながら`structuredContent`を返していない

これらを記事へ明記し、まとめで実装上の優先課題として扱った。

編集視点からは、対象読者と、二tool構成で何を優先したかも不足していたため追記した。

## 第3回評価

第2回と異なる技術編集と仕様監査の二つの視点で再評価し、93点と91点だった。

いずれも90点以上で、すべての評価軸が配点の60%以上となった。

仕様監査で、`outputSchema`と実際のtool resultの不整合は`call_command`だけでなく`list_commands`にもあると判明したため、本文を訂正した。

天気APIは疎通確認には使えるものの、認証情報を集約する利点が伝わりにくい。

そこで、APIキーで保護されたHevyのトレーニング履歴をChatGPTから振り返る利用例へ置き換えた。

実データを含む画面は掲載せず、プロンプトと呼び出し経路で利用場面を示した。

## 最終確認

第3回の指摘を反映した後、同じ二つの視点で再採点し、94点と95点だった。

Hevyの利用例により、プロンプト、toolの呼び出し、APIキーをモデルへ渡さない効果が一続きで分かるようになった。

匿名化した実行結果を加える余地はあるものの、記事の中心は分析結果ではなく接続と認証の設計であるため、実データを載せなくても論証は成立している。

全評価軸が80%以上となり、新たな2点以上の改善事項も確認されなかったため、今回のループはここで終了する。
